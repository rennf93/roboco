## Design Philosophy

The draft-review panel is the critical trust surface of Prompter. If users cannot distinguish between what the LLM guessed and what they explicitly asked for, they will lose trust in the system. If the confirmation step is too easy to skip, garbage tasks will enter the dev pipeline.

**Two principles govern this document:**

1. **Radical transparency.** Every AI-suggested field must be visually labeled, color-coded, and accompanied by an explanatory tooltip. Human-edited fields must also be labeled so the user sees their own impact.
2. **Friction by design.** The confirmation gate intentionally introduces friction — not to annoy, but to protect. It is a deliberate pause that says: "Stop. Read. Decide."

## Draft-Review Panel Layout

The panel is a full-viewport card that replaces the chat input area when the user enters the REVIEWING state. It is not a modal (the user can still scroll the chat context above), but it dominates the lower portion of the screen to signal that the interaction mode has shifted from "conversation" to "formal review."

## AI-Suggestion Affordances

### The AI Badge

Every field that the LLM populated receives an `[AI]` badge next to its label. This badge is **not** a `Badge` component (too large for inline use). It is a compact, accessible inline indicator:

```tsx
// AI-suggestion indicator component
<span className="inline-flex items-center gap-1 text-xs text-muted-foreground ml-2">
  <Sparkles className="h-3 w-3 text-primary" />
  <span>AI-suggested</span>
</span>
```

**Behavior:**
- On hover: `Tooltip` appears with text: "This field was generated by the AI based on your conversation. You can edit it."
- On focus (keyboard): same tooltip appears via `title` attribute fallback.

### The Human-Edited Badge

When the user edits an AI-suggested field, the `[AI]` badge transitions to a `[Edited]` badge:

```tsx
<span className="inline-flex items-center gap-1 text-xs text-amber-600 dark:text-amber-400 ml-2">
  <Pencil className="h-3 w-3" />
  <span>You edited this</span>
</span>
```

**Behavior:**
- Tracks `onChange` on the field input.
- Compares current value to original LLM-suggested value.
- If different → show `[Edited]`.
- If user reverts to original → revert to `[AI]`.
- If field was empty before LLM suggestion and user edits → still `[Edited]`, not `[AI]`.

### Never-Edited State

Fields that the user has not touched remain in `[AI]` state throughout REVIEWING and CONFIRMING.

### Color Contrast Compliance

The AI badge uses `text-muted-foreground` (OKLCH 0.556 0 0 on light, 0.708 0 0 on dark). The Edited badge uses `text-amber-600` / `dark:text-amber-400`. Both pass WCAG AA against the `background` color.

**Important:** The badge is **never** the sole differentiator. It is always accompanied by an icon (`Sparkles` or `Pencil`) and text label. This satisfies WCAG 1.4.1 (Use of Color).

## Field-Level Specifications

The draft-review panel exposes the same fields as `create-task-dialog.tsx`, organized into logical groups. Each field inherits the validation rules from the existing dialog.

### Group 1: Identity (Always Visible)

| Field | Component | AI-Suggested? | Validation |
|-------|-----------|---------------|------------|
| Title | `Input` | Yes | Required, 5–200 chars |
| Description | `MarkdownEditor` | Yes | Required, ≥20 chars |
| Team | `Select` | Yes | Required |
| Acceptance Criteria | `AcceptanceCriteriaEditor` | Yes | Required, ≥1 item |

### Group 2: Planning (Always Visible)

| Field | Component | AI-Suggested? | Validation |
|-------|-----------|---------------|------------|
| Priority | `Select` (P0–P3) | Yes | Defaults to P2 |
| Complexity | `Select` (Low/Med/High) | Yes | Defaults to Medium |
| Nature | `Select` (Technical/Non-Technical) | Yes | Defaults to Technical |
| Status | `Select` (Pending/Backlog) | Yes | Defaults to Pending |
| Task Type | `Select` (Code/Doc/Research/Planning/Design/Admin) | Yes | Defaults to Code |

### Group 3: Context (Advanced, Collapsible)

| Field | Component | AI-Suggested? | Validation |
|-------|-----------|---------------|------------|
| Project | `ProjectSelector` | No (user must pick) | Required unless Product is set |
| Product | `Select` | No | Optional |
| Parent Task | `TaskSelector` | No | Optional |
| Assign To | `AgentSelector` | No | Optional |
| Dependencies | `DependencySelector` | No | Optional |

**Rationale for AI-suggestion scope:**
- The LLM can reasonably infer title, description, acceptance criteria, team, priority, and complexity from a conversation.
- Project, product, parent task, assignee, and dependencies require organizational knowledge that the LLM does not reliably have. These remain manual fields to prevent incorrect routing.

### Inline Editing Behavior

All fields in Group 1 and Group 2 are **inline editable** — no "edit mode" toggle. The user clicks into the field and changes the value immediately, just like `create-task-dialog.tsx`. The `[AI]` → `[Edited]` transition happens instantly on `onChange`.

For fields using rich components (`MarkdownEditor`, `AcceptanceCriteriaEditor`), the component's existing `onChange` handler drives the badge transition.

## The Confirmation Gate

### Why a Gate, Not a Banner

A confirmation banner or toast at the bottom of the screen is dismissible by reflex. A checkbox next to the submit button is easy to miss. The gate must be a **modal interruption** that forces the user to context-switch from "editing" to "deciding."

### Gate Variants

Two variants are specified. The **primary variant** (Modal Gate) is the default. The **secondary variant** (Inline Review Banner) is available for power users who opt out of the modal in Settings.

#### Variant A: Modal Gate (Default)

As documented in [`state-machine.md`](state-machine.md#5-confirming--explicit-human-confirmation-gate):

- Full `AlertDialog` with `AlertDialogContent` at `sm:max-w-lg` (desktop) and `max-w-[calc(100%-2rem)]` (mobile).
- Read-only task summary inside the dialog.
- Three mandatory checkboxes.
- Primary action disabled until all checkboxes are checked.

#### Variant B: Inline Review Banner (Power-User Opt-In)

When enabled in user settings:

- No modal. Instead, a sticky banner appears at the top of the draft-review panel when the user clicks "Looks Good."
- Banner contains:
  - Warning icon + "You are about to create a task."
  - The same three checkboxes, now inline.
  - "Confirm & Create" button (disabled until checked).
  - "Cancel" button to dismiss the banner and return to editing.

**Rationale for variant B:** Power users who create many tasks per day may find the modal fatiguing. The inline variant preserves the checklist friction but removes the modality. It is **opt-in only** and defaults to off for all users.

### Gate Content (Both Variants)

**Checklist items:**

1. [ ] "I have reviewed the task title and description"  
   *Self-evidence: the user must have read the core identity fields.*
2. [ ] "I have checked the acceptance criteria"  
   *Self-evidence: the user must have verified that the criteria are testable and complete.*
3. [ ] "I understand this task will enter the dev pipeline and be claimable by agents"  
   *Self-evidence: the user acknowledges downstream impact.*

**Button labels:**
- Primary: "Create Task" (not "Submit" — "Create Task" is unambiguous)
- Secondary: "Go Back & Edit" (not "Cancel" — "Cancel" implies discarding work)

### Focus Management

On gate open (modal variant):
- Focus moves to the first unchecked checkbox.
- Tab order: Checkbox 1 → Checkbox 2 → Checkbox 3 → "Go Back & Edit" → "Create Task".
- Escape key closes the dialog (equivalent to "Go Back & Edit").
- Screen reader announces: "Confirmation dialog. Three items must be checked before you can create the task."

## Inline Editing Patterns

### Single-Line Fields (Title, Selects)

- `Input` or `Select` components as used in `create-task-dialog.tsx`.
- No "Save" button per field — changes are persisted to local component state immediately.
- Validation runs on blur and on "Looks Good" click.

### Multi-Line Fields (Description)

- `MarkdownEditor` component (reuses `panel/src/components/tasks/markdown-editor.tsx`).
- Toolbar appears on focus.
- `[AI]` / `[Edited]` badge sits above the editor, right-aligned.

### List Fields (Acceptance Criteria)

- `AcceptanceCriteriaEditor` component (reuses `panel/src/components/tasks/acceptance-criteria-editor.tsx`).
- Each criterion row shows its own `[AI]` / `[Edited]` badge.
- If the LLM suggests 3 criteria and the user edits 1, that row shows `[Edited]`; the other 2 remain `[AI]`.
- User can add new criteria (new rows show no badge until the user types, then `[Edited]`).
- User can delete AI-suggested criteria. Deleted criteria are removed from the draft but tracked in a "Removed Suggestions" collapsible section (so the user can restore them if they change their mind).

### Undo / Revert

Each AI-suggested field has a small "Revert to AI suggestion" button (ghost icon button with `RotateCcw` icon) that appears only when the field is in `[Edited]` state. Clicking it restores the original LLM value and reverts the badge to `[AI]`.

## Validation & Error Patterns

Validation rules are identical to `create-task-dialog.tsx`:

| Field | Rule | Error Message |
|-------|------|---------------|
| Title | ≥5 chars, ≤200 chars | "Title must be at least 5 characters" / "...less than 200 characters" |
| Description | ≥20 chars | "Description must be at least 20 characters" |
| Acceptance Criteria | ≥1 item | "At least one acceptance criterion is required" |
| Project | Required unless Product set | "Pick a Project or a Product" |

**Error display:**
- Field border changes to `border-destructive`.
- `text-xs text-destructive` message appears below the field.
- If error exists, the `[Edited]` badge on that field gains a `text-destructive` color override to draw attention.

**Global error banner:**
If the user clicks "Looks Good" while validation errors exist:
- A `Alert` component (if available) or a `Card` with red left-border appears at the top of the panel:
  - "Please fix the highlighted fields before confirming."
- The panel auto-scrolls to the first invalid field.
- "Looks Good" button remains enabled (so the user can click it again after fixing), but the gate will not open until validation passes.

## Interaction Flow Example

1. User chats: "I need a login API with JWT and rate limiting."
2. LLM returns draft:
   - Title: "Build login API endpoint"
   - Description: "Create a RESTful login endpoint..."
   - Team: BACKEND
   - Priority: P1
   - Acceptance Criteria: ["Returns JWT on valid credentials", "Rate-limited to 5 req/min"]
3. Panel shows all fields with `[AI]` badges.
4. User edits title to "Build authenticated login API with JWT and rate limiting." → Title badge becomes `[Edited]`.
5. User adds a third acceptance criterion: "Invalid credentials return 401 with generic error message." → New row shows `[Edited]`.
6. User deletes the second AI-suggested criterion. → Row moves to "Removed Suggestions" section.
7. User clicks "Looks Good."
8. Validation passes.
9. **Modal Gate** opens.
10. User reads summary, checks all three checkboxes.
11. User clicks "Create Task."
12. Task created, redirect to detail page.
